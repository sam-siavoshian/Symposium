How Symposium works

When Claude encounters a task, Symposium spawns parallel research agents that each approach the problem from a different angle. One reads live documentation. One searches real GitHub repositories. One maps failure modes. One studies the existing codebase.

Findings get cross-validated using dual-mode research: both Oracle (docs) and Tracer (real code) run in parallel on the same dimension. When they agree, confidence goes up. When they disagree, that's a signal something is off.

Evidence scoring is computed from actual signals: agent convergence, unique repo count, cross-validation agreement, pattern consistency. Not an LLM guessing "I'm 85% confident." The verified plan is then delivered to Claude, and novel discoveries are stored so the next user gets an instant answer.

What this means

LLMs are great at the stuff in their training data. Add Stripe. Read docs. Update an API. But some tasks, they just fail. Hallucinate a function signature. Use a deprecated API. Confidently write broken code.

compile-compcert required knowledge about CompCert's build system that wasn't in Claude's training data. Vanilla Claude timed out at 900s trying to figure it out. Symposium researched it live, found the correct configuration, and Claude solved it in 1068s.

cancel-async-tasks needed specific async cancellation patterns. Vanilla gave up after 41s with a wrong approach. Symposium researched the patterns, gave Claude verified knowledge, and it solved it in 759s.

The gap between "what models know" and "what they need to know" is where Symposium lives. And every discovery becomes training data for better models tomorrow.
